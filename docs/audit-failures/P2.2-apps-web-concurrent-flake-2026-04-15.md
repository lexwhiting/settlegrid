## Symptom

## Why P2.2 surfaced it (wasn't failing pre-P2.2)

Post-P2.1 turbo runs were reliably green (verified earlier in this session after commits `ccb13a7` and `23fde03`). P2.2 added:

- 9 fast-glob + fsp-driven detection tests
- 9 source-resolver filesystem tests
- Multiple subprocess-spawning smoke tests in `index.test.ts`
- The `fast-glob` dependency (externalized, but adds load at test time)

These additions raised the total CPU + I/O footprint during the parallel-turbo window, pushing apps/web's already-tight 5s timeouts past the threshold.

## Mitigations applied on the P2.2 side

- `packages/settlegrid-cli/vitest.config.ts` â†’ `fileParallelism: false`. Serializes the cli package's 4 test files so they no longer compete internally, reducing peak load during the turbo window.
- `packages/settlegrid-cli/src/index.test.ts` â†’ smoke tests consolidated from 7 â†’ 5 (9 spawns â†’ 8). Cuts subprocess startup load.
- `scanSourcesFor` in `detect/index.ts` sets `followSymbolicLinks: false` and enforces 500-file / 5MB / 10s caps. Independent robustness; not specifically for the flake but keeps scan bounded.

These reduce flake frequency (observed ~40% after mitigation) but don't eliminate it â€” the root cause is on the apps/web side.

## Root-cause fix (out of P2.2 scope)

`apps/web` spec explicitly listed under "Files you MUST NOT touch" for P2.2, so the fix must land in a follow-up scoped to apps/web:

1. Raise `testTimeout` for the two dynamic-import tests (e.g. to 15s) â€” localized fix.
2. **OR** prewarm `@/lib/settlement` in a `beforeAll`/`beforeAll` at the file level so dynamic imports aren't racing transform.
3. **OR** convert the two tests from `await import(...)` to top-level static imports, moving the transform cost out of the test body.

Option 2 is probably the lightest-touch and most correct.
